# План реализации E2E шифрования с синхронизацией ключей

## Текущее состояние

Уже реализовано в текущем рефакторинге:
- ✅ CSRF токен (защита от CSRF-атак)
- ✅ `POST /api/auth/update-public-key` — отправка публичного ключа на сервер
- ✅ `GET /api/chats/:chat_id/keys` — получение ключей участников чата
- ✅ Генерация NaCl ключей на клиенте
- ✅ Шифрование/дешифрование сообщений (частично)

## Архитектура

### 1. Хранение токенов в IndexedDB (авто-шифрование без PIN)

**Проблема:** токены в IndexedDB должны быть зашифрованы, но без ввода PIN.

**Решение:** шифровать токены **приватным ключом NaCl** пользователя.

```
Приватный ключ NaCl (есть на устройстве)
    ↓ (используется как symmetric key)
Зашифрованные токены → IndexedDB
```

**Как работает:**
1. При входе генерируется/получается приватный ключ NaCl
2. Токены (access, refresh) шифруются публичным ключом NaCl (self-encryption)
3. Зашифрованные токены сохраняются в IndexedDB
4. При чтении — расшифровываются локально приватным ключом NaCl

**Свойства:**
- На этом устройстве — токены работают автоматически
- На другом устройстве — токены бесполезны без приватного ключа
- Сервер не может расшифровать токены
- Если украдут IndexedDB — токены зашифрованы

### 2. Синхронизация E2E ключей через подтверждение устройством

**Проблема:** на новом устройстве нет приватного ключа — старые сообщения недоступны.

**Решение:** передача ключа с подтверждением от первого устройства.

```
Новое устройство (вход)
    ↓ Запрос "нужен ключ"
Сервер
    ↓ Push-уведомление
Первое устройство
    ↓ "Подтвердить вход?"
Пользователь → Подтверждает
    ↓ Зашифрованный blob
Новое устройство → Расшифровывает локально
```

**Свойства:**
- Сервер видит только зашифрованный blob
- Без подтверждения ключ не передаётся
- Пользователь просто нажимает "подтвердить" — прозрачно

### 3. Опциональный пароль (backup)

**Когда нужен:** первое устройство недоступно (потеряно, сломано).

**Как работает:**
- При настройке: пользователь может установить пароль (или не устанавливать)
- Пароль используется как дополнительный ключ шифрования приватного ключа NaCl
- На новом устройстве: вводишь email + код + TOTP + пароль (если установлен)

## Реализация

### Этап 1: Откатить PIN-логику (оставить CSRF, E2E)

**Изменения:**
1. `web-client/src/components/PinSetup.vue` — удалить файл
2. `web-client/src/crypto/index.ts` — убрать `getUserPin()`, PIN-шифрование
3. `web-client/src/db/index.ts` — вернуть хранение токенов открытыми (или шифровать ключом NaCl — см. этап 2)
4. Убрать из .gitignore упоминание о user_pin (если есть)

### Этап 2: Шифрование токенов ключом NaCl

**Сервер:** изменений не требуется (сервер не хранит токены)

**Клиент:**
1. `crypto/index.ts`: добавить функции `encryptWithKeypair`, `decryptWithKeypair`
2. `db/index.ts`: изменить `saveAuth`, `getAuth` — шифровать токены перед сохранением

**Схема:**
```
Токен (access/refresh)
    ↓ nacl.box.after (публичный ключ получателя = публичный ключ пользователя)
Зашифрованный blob → IndexedDB
```

### Этап 3: Серверная часть синхронизации ключей

**База данных (миграция):**
```sql
-- Хранение зашифрованного приватного ключа
ALTER TABLE users ADD COLUMN encrypted_private_key BYTEA;

-- Запросы на синхронизацию (одно устройство подтверждает другое)
CREATE TABLE key_sync_requests (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    device_name VARCHAR(100),
    code VARCHAR(20),
    status VARCHAR(20) DEFAULT 'pending', -- pending, approved, rejected
    created_at TIMESTAMPTZ DEFAULT NOW(),
    expires_at TIMESTAMPTZ DEFAULT NOW() + INTERVAL '10 minutes'
);
```

**API:**
- `POST /api/keys/request-sync` — запросить синхронизацию (генерирует код)
- `POST /api/keys/approve-sync` — подтвердить с первого устройства (кодом)
- `GET /api/keys/download` — скачать зашифрованный ключ после подтверждения
- `POST /api/keys/upload` — загрузить ключ на сервер

### Этап 4: Клиентская часть синхронизации

**Сценарий 1: Первое устройство (основное)**
1. При входе проверить: есть ли `encrypted_private_key` на сервере
2. Если нет — сгенерировать ключи NaCl, зашифровать публичным ключом (self), загрузить на сервер

**Сценарий 2: Новое устройство**
1. Вход (email + код + TOTP)
2. Запрос ключа с сервера
3. Сервер отправляет код на email (или генерирует ссылку)
4. Пользователь вводит код на первом устройстве
5. Сервер подтверждает, передаёт blob на новое устройство
6. Новое устройство расшифровывает локально

**Сценарий 3: С опциональным паролем**
1. Вместо подтверждения — ввод пароля
2. Ключ зашифрован паролем + публичным ключом
3. Новое устройство расшифровывает: пароль → ключ → ключ NaCl

---

## Ограничения

1. **Без подтверждения:** старые сообщения недоступны
2. **Опциональный пароль:** не обязателен, но рекомендуется для backup
3. **Одно устройство:** если единственное устройство потеряно — ключ потерян безвозвратно (нужна ручная backup)

---

## Безопасность

- Сервер **не может** расшифровать ключи (видит только blob)
- Токены в IndexedDB зашифрованы (без ключа бесполезны)
- CSRF защита на всех запросах
- 2FA уже требуется для входа
